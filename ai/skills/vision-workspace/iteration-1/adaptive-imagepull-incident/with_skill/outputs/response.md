**Conclusion:** Most likely cause: the service account's `imagePullSecret` is stale relative to the registry's credential rotation/expiry policy, producing intermittent `401 Unauthorized` on pull even though push (a separate credential path) still succeeds. Recommend verifying this hypothesis with a short, ordered check before making any change.

**Scope and approach:** Started with a general incident hypothesis rather than launching a full specialist panel — the evidence (push succeeds, pull returns 401, no prod changes, one aging credential) already points strongly at one mechanism. This is adaptive expansion: begin narrow, and only add a targeted specialist if verification disconfirms the leading hypothesis. No build, broad deployment-config, or observability-platform workstream was launched, since none of that evidence surface is implicated by a pull-side 401.

**Evidence-backed finding:** Push and pull to/from a registry commonly use different credentials/scopes (a CI/build identity for push vs. a cluster service-account identity for pull). A healthy push with a failing, intermittent pull-side 401 is inconsistent with a broad registry outage or network issue (which would typically fail both, or fail consistently) and is consistent with a single stale credential on the pull side — especially given the imagePullSecret is 6 months old and no prod change coincides with the onset. `ImagePullBackOff` is Kubernetes' generic surfacing of any pull failure; the `401` in the runtime's own report is the more specific signal and points at authentication, not image absence or network reachability. The `helm --wait` timeout is a downstream symptom (pods never reach Ready) and not itself informative about root cause.

**Ordered verification steps (in priority order):**
1. Check registry-side auth logs for `401`/credential-rejection events correlated in time with the failing pulls — this converts "likely" into "confirmed" or "disconfirmed."
2. Compare the imagePullSecret's embedded credential/token expiry (or the registry's stated rotation policy) against its 6-month age — if the registry enforces a shorter rotation window, this is very likely the root cause.
3. If steps 1–2 confirm: regenerate and redeploy the imagePullSecret, and re-verify pulls succeed across all affected nodes (intermittent, not constant, failure can mean partial secret propagation — check this before closing the incident).
4. If steps 1–2 do not confirm (e.g., auth logs show no rejections, or the credential is still valid): expand to a targeted network/registry-availability specialist, since the "intermittent" pattern would then be more consistent with transient registry-side throttling or DNS/network flakiness between specific nodes and the registry.

**Assumptions / unresolved questions:** Assumes the registry enforces some form of credential expiry/rotation (common for ECR tokens, private registry PATs); if the registry uses long-lived static credentials, the stale-secret hypothesis weakens and step 4's expansion becomes the more likely path. Whether the failures cluster on specific nodes (partial propagation) or are uniformly random across the fleet is not yet known and should be checked in step 3 regardless of which branch confirms.
