**Evidence-backed finding:** All 6 failures (20% of 30 runs) occur during `npm ci` with `ECONNRESET`, and the lockfile is identical between passing and failing runs. `ECONNRESET` is a network-layer connection reset, not an npm dependency-resolution error — if the lockfile were the cause, failures would need a different lockfile or a version-conflict error, neither of which is present. A consistent lockfile across both outcomes rules out dependency drift as an explanation and leaves transient network/registry connectivity as the best-supported cause. No retry is currently configured, so any single transient reset fails the entire job with no self-recovery.

**Proportionate next steps:**
1. Add a bounded retry (2–3 attempts with short backoff) specifically around the `npm ci` step — this directly targets a transient-network hypothesis without requiring infrastructure access this runtime doesn't have.
